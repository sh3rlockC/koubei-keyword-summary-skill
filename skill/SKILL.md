# 口碑关键词与摘要提炼

目标：读取已有口碑 Excel，提炼高频关键词、方向摘要、代表性原句，并导出适合汇报的摘要文件。

## 1. 支持的输入

优先支持两种模式：

### A. 单平台模式（汽车之家）

输入一个已导出的汽车之家口碑 Excel。

适用结构：
- 单 sheet / 多 sheet
- 但必须包含 `最满意` 与 `最不满意` 两列
- 当前上游采集层通常输出 `购车口碑` / `试驾口碑` 两个 sheet，但本脚本只依赖列契约，不依赖 sheet 名

### B. 双平台整合模式（默认推荐）

同时输入：
- 汽车之家原始口碑 Excel
- 懂车帝原始口碑 Excel

当前默认目标：
- **最终直接输出双平台整合摘要 Excel**
- 自动包含：总览、跨平台对比、业务摘要、产品机会点、一页纸总结

如果用户明确说“整理汽车之家+懂车帝口碑”“双平台汇总”“整合成一个表”，优先走双平台模式。

## 2. 核心提炼目标

至少输出四层结果：

1. **关键词频次 / 方向统计**
   - 优先统计业务可读的方向词，不追求学术分词
   - 合并同义表达，例如：
     - `空间大` / `空间宽敞` → `空间`
     - `车机卡` / `软件不好用` / `应用太少` → `车机智能化`

2. **方向级摘要**
   - 每个方向输出：
     - 提及次数
     - 简短摘要
     - 代表性原句 3-5 条

3. **总览 / 业务摘要**
   - 输出适合直接发给业务、产品、老板看的短口径
   - 至少包含：
     - 核心卖点
     - 核心槽点
     - 产品建议
     - 适合人群 / 使用场景

4. **一页纸总结**
   - 放在最终 Excel 最前面
   - 控制在一屏内能扫完
   - 包含：
     - 样本基础
     - 一句话结论
     - 核心优点 Top5
     - 核心槽点 Top5
     - 平台差异观察
     - 汇报口径建议

## 3. 双平台模式的处理规则

### 汽车之家

直接按 `最满意 / 最不满意` 结构提炼：
- 高频方向
- 高频词
- 方向摘要
- 代表性原句

### 懂车帝

懂车帝原始表通常只有 `评价全文`，没有天然的 `最满意 / 最不满意` 双列。

当前默认做法：
- 从 `评价全文` 中提取“优点 / 满意 / 最满意 / 好评”相关句子，作为正向样本
- 提取“缺点 / 不满意 / 最不满意 / 槽点”相关句子，作为负向样本
- 如果正文没有显式分段，再用轻量规则从全文中兜底识别正向/负向句子
- 最终输出：
  - `懂车帝_正向摘要`
  - `懂车帝_负向摘要`

注意：
- 不要伪装成“精确情感分析”
- 默认按“业务可读、能汇报”优先
- 若长评本身已经明确写出优缺点段落，优先相信原文结构

## 4. 默认输出结构

### 单平台模式（汽车之家）

默认输出：
- `总览摘要`
- `业务摘要`
- `产品机会点`
- `最满意方向统计`
- `最不满意方向统计`
- `最满意方向摘要`
- `最不满意方向摘要`
- `最满意主题`
- `最不满意主题`
- `最满意高频词`
- `最不满意高频词`

### 双平台整合模式

默认输出一个 Excel，建议包含：
- `一页纸总结`
- `总览摘要`
- `跨平台对比`
- `综合业务摘要`
- `产品机会点`
- `汽车之家_满意摘要`
- `汽车之家_不满意摘要`
- `懂车帝_正向摘要`
- `懂车帝_负向摘要`

同时会在同名路径旁自动生成 `.validation.json` 校验报告，记录输入检查、输出检查、告警与错误。

默认命名示例：
- `启源A06_双平台口碑摘要.xlsx`

## 5. 抽取原则

- 不改写用户原句的事实含义
- 关键词提炼优先服务业务阅读
- 同义归并要稳定、一致
- 代表性原句保留原文口语感
- 如果某方向只出现 1-2 次，可以保留，但不要夸大成主流结论
- 双平台整合时，优先给“共同结论”，不要只堆两份表

## 6. 推荐执行方式

使用附带脚本：
- `scripts/summarize_koubei_excel.py`

### 单平台示例

```bash
python3 skills/koubei-keyword-summary/scripts/summarize_koubei_excel.py \
  --input /path/to/ZJ口碑_启源A06.xlsx \
  --output /path/to/启源A06_汽车之家口碑摘要.xlsx \
  --model-name 启源A06
```

### 双平台整合示例

```bash
python3 skills/koubei-keyword-summary/scripts/summarize_koubei_excel.py \
  --autohome-input /path/to/ZJ口碑_启源A06.xlsx \
  --dcd-input /path/to/DCD口碑_启源A06.xlsx \
  --output /path/to/启源A06_双平台口碑摘要.xlsx \
  --model-name 启源A06
```

### 进度条

- 默认在交互终端自动显示进度条
- 如果要强制显示，可加 `--progress`
- 进度条覆盖输入读取、汽车之家方向统计、懂车帝长评拆分等长循环
- 如果要给飞书或对话框查询，可额外加 `--progress-file`，脚本会持续写同名 JSON 进度文件，建议前端或对话框服务每 1-2 秒轮询一次
- 如果要直接推送到通用 webhook，可加 `--progress-webhook`
- 如果要直接推送到飞书 incoming webhook，可加 `--feishu-webhook`，如机器人启用了签名再加 `--feishu-secret`
- `--progress-file`、`--progress-webhook`、`--feishu-webhook` 可以同时开，文件适合轮询，webhook 适合主动推送

#### 轮询示例

后端收到任务后，把 `--progress-file /tmp/job.progress.json` 传给脚本；对话框端定时读取这个文件，然后把 `percent`、`stage`、`message` 显示出来。

前端轮询示例：

```javascript
setInterval(async () => {
  const res = await fetch(`/api/jobs/${jobId}/progress`);
  const data = await res.json();
  setProgress(data.percent);
  setStage(data.stage);
  setDetail(data.message);
}, 2000);
```

后端接口只要把进度文件内容原样返回即可：

```python
@app.get("/api/jobs/{job_id}/progress")
def job_progress(job_id):
    return json.loads(Path(progress_path).read_text(encoding="utf-8"))
```

#### Feishu incoming webhook 示例

脚本会把进度转换成飞书机器人可接收的文本消息，不需要你自己拼消息格式。
如果机器人配置了安全密钥，脚本会自动附带 `timestamp` 和 `sign`。

## 7. 校验要求

导出前至少检查：

### 单平台模式
- 是否识别到 `最满意` 和 `最不满意` 两列
- 是否成功读取全部 sheet
- 关键词统计非空
- 总览摘要非空

### 双平台模式
- 汽车之家输入是否能识别 `最满意` / `最不满意`
- 懂车帝输入是否包含 `评价全文`
- 双平台至少有一侧提炼出有效方向统计
- 最终 Excel 是否成功生成 `一页纸总结`

如果原始 Excel 缺列或为空，直接报错，不输出伪结果。

### 自动校验输出

脚本会自动生成同名 `.validation.json`，其中包含：

- 输入文件检查结果
- 识别到的 sheet 和行数
- 输出 sheet 的行数和列名
- `warnings` 与 `errors`

默认情况下，输入缺列、缺 sheet、关键输出为空时会直接失败。
如需把告警也视为失败，可加 `--strict-validate`。
